DaoCloud介绍

国内众多提供Docker服务的一员，最开始提供免费的高速镜像，但后来速度越来越不行而且还必须用他们的工具。（单纯加速可以用这个了 https://dev.aliyun.com/）

功能

  • 通过git自动build docker， build的时候可以先跑个集成测试。
  • build后可以放到私有的仓库。
  • 不论是公有的docker还是私有的都可以直接运行在所谓的集群中，其实就是docker的宿主。可以使用daocloud的宿主，也可以用自己的主机。
  • stack，data Volume什么的就和tutum差不多，用过tutum或者cloud.docker就会有非常熟悉的感觉。

我的使用流程

  1. 当然是弄个Dockerfile, 配置一下运行环境，设置端口和启动程序。
  2. 写业务代码，和Docker的其他相关文件放到一个git里面。
  3. push 到github，DaoCloud会自动根据dockerfile build。
  4. build成功或者失败都可以微信通知我（赞）。
  5. build成功后自动Push到自己的私有仓库，生成一个新的版本。
  6. 这个daoker可以部署成一个应用，应用的运行环境可以选择自己的主机或者DaoCloud提供的docker环境。1x（128mb）~ 16x，价格和网速还可以比较省心。我的爬虫就用的4x，每个月大约几十块。
  7. 若已经由应用在使用这个docker，则当有新版本的时候可以自动被部署。

注：

  • 我写的有点儿乱，但是用过docker的人应该明白。
  • 整个过程很流畅，我写好业务代码，会用babel转化到docker的目录。提交git后就不用管了，都是自动的。

基于leancloud的全栈开发流程

接上一篇 一个React+leancloud+daoCloud全栈项目的程序结构

leancloud是什么？

一个称为BaaS（Backend as a Service）的平台，其中综合了很多互联网项目常用的功能，为开发者提供了非常多的方便。我之前用过parse，所以用leanclou还是很熟悉的。

功能

  1. LeanStorage：

    • 相当于一个只提供restAPI的mongodb，但同时有丰富的SDK可以使用。
    • 按照api请求次数收费，每月免费100万次。
    • 优点1：免维护，开箱即用，自动处理备份，读取缓存等功能。
    • 优点2：有一个应该是基于spark的离线检索功能，使统计数据成为可能。
    • 优点3：由较为完整的api请求统计及访问日志，debug和优化项目的时候很有用。
    • 优点4：功能完善，例如后台，导出导入，多app共享class等功能。
    • 特点1：有完整的权限系统，但要想控制好权限需要下很多功夫。
    • 特点2：文件实际是放在七牛的但开发者完全感觉不到，不过直观感觉挺慢的。
    • 缺点1：1万条数据后无法创建索引。

  2. LeanCache：

    • 就是redis，可以用作提高并发，较少api请求。
    • 价格便宜

  3. LeanEngine：

    • 应该是定制化的docker，支持node，python，php（貌似还有别的）
    • 提供两个环境，生产环境和stage环境，能够实现强制的健康检查和无缝切换。
    • 相当于自带了负载均衡。
    • 优点：自带的cli工具发布调试很方便。
    • 缺点：cname方式的域名绑定导致顶级域名和邮箱的解析冲突。

  4. LeanSession(我自己起的名字）

  • 简化了所有通用的用户相关的东西，包括注册，登录，找密码，邮箱验证，短信验证，三方登录等。
  • 缺点也很明显，和leancloud绑定的很紧，未来想解绑比较难

优点

  • 节省了大量的时间，web服务器，数据库，缓存都是免维护的。

缺点

  • 上手容易，但这要弄懂还是很别扭的。
  • 稳定行一般，偶尔会出问题。

LeanCloud架构的理解

理解起来有难度，我也未必能说清楚，尽量试试吧：

  • 云端的leanEngine环境分prod和stag两个环境，

    • web访问的时候可以通过二级域名（xx.leanapp.cn）前面加stag-来区分。
    • 但api访问需要在http header中加标签才行。
    • 这里需要特别注意一下，你从stag-xx.leanapp.cn访问了网站，返回的html是stag环境的，但是stag环境html中的jsSDK依然访问的是正式prod环境的云函数。

  • leanStage只有一个环境，通过任何渠道访问和修改数据都是改的一个数据。

  • 通过lean up可以启动一个本地leanengine环境。该环境也分两种：
    • localhost:3000 返回本地渲染的html页面，但需要特别注意的是html中的JS SDK访问的是云端prod环境的leanengin，和本地毫无关系。
    • localhost:3001 返回一个本地的cloud函数和hook函数的调试页面，此处访问的是本地环境的leanengine函数，但数据依旧是来自云端的lenastroage。

开发流程

需要的东西

  • 所有的包都是npm中的，所以用webpack打包异常方便。
  • webpack的dev server作为本地开发服务器，
  • hot-loader：但实际使用后没有那么神奇，源码改变后一般需要自动刷新页面。
  • bater-npm-run: 主要是简化环境变量的配置
  • babel: 重头戏，主要是presets ES2015
  • gulp: 用于babel编译，和发布内容打包等。

基本流程

  1. 先开发服务端的cloud函数，然后通过gulp编译到dist目录。通过lean up调试。
  2. 服务端调试ok了就，直接发布到leancloud的生产环境。
  3. 开发web端，主要是通过webpack的server。只是可以调用第一步发布的cloud函数了。
  4. web调试OK了，就用webpack编译到dist目录。
  5. 通过gulp打包整体的server文件到dist目录，同时吧web代码也用babel编译到dist目录中用于服务端渲染。
  6. lean up调试后发布到stag环境。
  7. stag环境没问题后直接发不到prod环境，基本没有任何间断。

可能的问题

  1. CDN的问题：所有的js，css其实是在七牛上有缓存，但每次发布都使用md5命名js和css文件，所以就解决了cdn更新的问题。
  2. 缓存问题：发布到prod环境的时候需要进行健康检查（就是发起一个特定的get请求），我就在这个请求中清空了必要的缓存。
  3. SDK的问题，浏览器中用的是lean-storage SDK，Node环境用的是leanengine SDK，两者略有却别。若混用则会造成用户登录状态混乱的问题。主要是服务端渲染的时候可能需要引用SDK运行云函数，最好环境判断就好了。

关于单元测试

  1. 开始所有的cloud请求是加了单元测试，但后来不弄了。因为：
    • leancloud还很稳定，其实不需要测试。可以假设他一直是可靠的。
    • 部署测试数据，然后测试，然后清除测试数据太麻烦了。
  2. Server端的models开始是写测试的，但覆盖率不高。
  3. web端测试是mock了一些cloud函数，然后sinon注入进去，进行测试。后来放弃了：
    • 没法完全mock所有cloud函数，开发环境中还是连接的远端leancloud。
    • 自动单元测试只是测试action和reducers，感觉意义不大。

综上所述，最终只是做了很简单的测试。以后代码再稳定一些的时候再加吧。

关于集成测试

  1. codeship：其实不错，但我用的时候不是docker环境。
  2. daoCloud集成测试：优点是部署环境是国内访问leancloud方便。node_modules没有缓存，所以非常缓慢，几乎没法用。
  3. CircleCI：非常完美，而且基础项目免费。但是服务器在国外，方位leancloud非常慢，经常由于timeout而failed。我尝试过复制一个lenacloud的美国节点项目用于测试，但是操作起来比较麻烦，所以我集成测试的时候就跳过了leancloud的依赖。

关于开发环境

  • 一个app只有一套LeanStorage, 所以没法在一个app中区分开发环境和生产环境。不过创建一个新的app非常简单，所以可以用两个app一个试生产环境一个是开发环境。
  • 但是隔一段时间将生产环境的数据导入到开发环境，虽然提供了导入导出功能，但必须一个class一个class的导入，比较麻烦，目前还没有找到合适的方法。

一个React+leancloud+daoCloud全栈项目的程序结构

2016-09-19

有一个挺热的词儿叫SaaS（Software as a Service），本来是指一种按需购买而不是一次性买断的软件购买方式。但随着技术的发展不仅商业软件大量采用了SaaS的方式，很多原本的服务也都开始像软件一样在互联网上售卖。就好像SaaS写成了Service as a Software。

举个栗子，一个初创的互联网公司，可能需要云主机，数据库软件，测试服务，数据统计服务，软件集成，众包开发，交互设计，客户服务，客户CRM，广告投放，人员管理等方方面面的服务。而目前所有的服务都可以通过互联网购买。据国外的组织粗略统计平均每个初创公司要使用60个网络服务。

如此多的服务分散在网络中很难被人发现，所有我将它们集合到一个统一的目录中并做好分类写好描述。这个项目的名字就叫目录木（mulumu.com）。目录木的基本逻辑就是：

1. 由编辑（我本人）搜集SaaS服务并添加好分类和描述。
2. 抓去互联网公司的使用了那些服务（主要是和web，domain，服务器相关的）并建立真实可信的使用关系数据。
3. 互联网公司的员工可以主动的注册目录木并通过公司邮箱认证，认证后主动标记自己再用的SaaS服务。
4. 所有人都可以浏览大部分数据，通过真实的使用数据来找到合适自己的SaaS服务。
5. 如果说的不明白，自己看看网站就可以了。

目录木大致的模块有

  1. react前端
  2. node后端，运行在leancloud上。
  3. node小爬虫抓取企业数据，运行到daoCloud提供的docker容器内。

目录木本身是一个由JS写成的全栈项目，采用了大量的SaaS服务，相交于一些网上的JS全栈例子还是要复杂和全面一些。我可以一个人短时间开发出来全靠开源软件和优秀的云服务，我想通过一系列文章分享一下目录木所使用的技术和服务。

如果感兴趣请注意，

1. 技术有限，高手绕过。
2. 并非入门教程，不会说明具体细节也不会开放源代码。
3. 这是一个有广告意图的文章，不喜勿喷，看后请关注目录木，谢谢。

1.技术栈，服务栈

用了什么可以先看这个 http://www.mulumu.com/company/mulumu

用什么：

  • React及相关技术，如Redux，Webpack等
  • AntDesign：单独列出来是因为它极大的丰富了web控件，React结合这种完整的Component库才能得以发挥。
  • leanengine: 实际是基于Docker，添加了负载均衡，添加了生产环境和与发布环境的切换，依赖缓存，cli工具等附加功能。价格适中。
  • leanstorage：基于云的nosql，提供部分hive SQL功能。100w请求免费。
  • leanCache：就是redis，与leancloud服务配合使用价格适中。
  • daoCloud服务：用于目录木爬虫程序的自动化测试+自动化部署+自动运行。免费4x计算单元，基本免费。
  • 日志易：基本上就是elasticsearch， 免费存7天数据，用于记录目录木爬虫的日志和监控异常。
  • daovoice：山寨intercom，不过体验太差，intercom降价了正在考虑换。
  • 其他项目标配：github, GA，百度统计，监控宝，阿里域名……

不用什么：

  • JQuery及相关的前段库：1. 因为和React不是很好融合。 2. AndD的控件基本够用。3. 不考虑兼容问题，目录木兼容ie11以上。
  • mongodb，mysql等数据库：用leanstorage免维护，免备份，节省了大量时间，同时也逼迫自己优化程序减少调用，因为每次调用都花钱。
  • 阿里云等vps：daocloud，leancloud实际上也是基于aliyun的，不直接使用主要是为了省时间，在项目较小的时候使用阿里云上层服务反而比较省钱。
  • 灵雀云，时速运等docker服务：至少我体验时他们都像个半成品。
  • OneApm：比NewRelic贵。
